NextJS - Supabase - Prisma

By Ed Anisko

February 24, 2019

** Making a Free NextJS Front-end, SQL back-end with Migrations **

Is PHP Dead? It’s everywhere on the internet. Didn’t the know it all managers learn their lesson from Ruby on Rails. It’s gone. Dead. Latin. Spotify and Netflix are using it. But they are the only companies that can afford to hire a Ruby dev.

Ruby was a flash in the pan. I did like Rails. Which Laravel stole from heavily and I love me some Laravel. Now what? pnpm, bun, Rust with WebAssembly? I want easy. Laravel easy. But the Javascript frameworks are basically just figuring out server server side rendering. No SQL databases seam fine until you have to count how many rows you have. That’s when you need a real database. Even Ruby on Rails knew enough to stay away from No SQL.

But here we are. Am I really going to build my next big idea on Laravel. There are no PHP lambda’s. So that answers it. No.

But I want things. I want SQL because, stop the madness. I want ORM so that my tables and my objects understand each other. Migrations in case I change something and want to continue testing my beuatiful, magical idea with some degree of hope that it will continue to work.

I’ve been reading about the Javascript Framework bubble for a while now. There is no decernable winner to be sure. Angular looked like a winner, then React. They both changed so fast and broke themselves. And they were really frameworks. They were, PHP3. The space has matured a bit. It feels like PHP 5. Back when MVC was a thing. The wheel keeps on turning. Giddy up.

Javascript here we come. Mac people go install brew. Windows people go borrow a Mac.

Like the title says. NextJS, Supabase and Prisma. NextJS which is React with a built in router and a sane way to structure your code base. Supabase is a Google Firebase alternative. In truth it could stand on its own as an AWS RDS replacement the little projects that I start here and there. The paid tier is $25 a month. Which is better than running AWS Multi AZ Aurora with 2 t3-smalls. I usually add elastic cache in there too. Which by the end of it all runs at about 200/month for a little idea. Too Much. I need to be part of the everything is free until you’re successful party. Supabase looks like its trying to give me that. I will take it. Prisma is the ORM.

First NodeJS. It took at least five years to be able to install node and run npm without it blowing up. It’s stable now. You’re not suppose to use the odd numbered versions. I install NVM which is a node versin manager like pyenv. Install it with brew. Then install Node with NVM.

Now pick a package manager, you can use npm, yarn, bun or pnpm. These are to JavaScript as composer is to PHP. Why are there so many? idk. Node comes with npm but bad slow. PNPM is fast and caches things you’ve already downloaded then symlinks the node_modules directory to a central cache buried in your home directory. Bun is also fast but quirky for me. Brew install pnpm.

Here is the service-structure.

Vercel for Hosting Supabase for Database Supabase for Auth

First get yourself a github account. It’s free. Vercel and Supabase will allow you to sign in with your github account.

Let’s fire it up.

Head over to the command line and create a new directory.

pnpm create next-app –experimental-app

Keep saying yes until you have a new app installed in a directory named my-app.

And we are usuing TypeScript because we are warrior poets.

cd my-app

pnpm dev

You should see a new browser window open with a NextJS app. It’s a little boring. Let’s change that.

Tailwind can help with that. I bought all the templates so I may as well. Here’s how that goes.

pnpm install -D tailwindcss postcss autoprefixer

pnpx tailwindcss init -p

This will create your tailwind.config.js and postcss.config.js files. You will edit these so that when the JavaScript magic app compiler comes along and transpiles your code into brower friendly code it will also run the tailwind css compiler. This is a little confusing. But it’s not that bad. You will need to add a few lines to your postcss.config.js file. It should look like this.

/** @type {import('tailwindcss').Config} */
module.exports = {
  content: [
    "./app/**/*.{js,ts,jsx,tsx}",
    "./pages/**/*.{js,ts,jsx,tsx}",
    "./components/**/*.{js,ts,jsx,tsx}",
  ],
  theme: {
    extend: {},
  },
  plugins: [],
}

Then delete whatever is in you global.css file and relpace it with this.

@tailwind base;
@tailwind components;
@tailwind utilities;

Take a look in the app directory for a css file named page.module.css. Empty it out for now. This structured naming is the way NextJS wants you to do things. It makes keeping track of things easier. For now just empty it out.

Ther is a page.tsx file. This is the home page. Let’s skin it back to basics.

export default function Home() {
  return (
    <h1 className="text-3xl font-bold underline">
      Hello Next and Tailwind!
    </h1>
  )
}

pnpm dev

You should see the hello message with Tailwind classes styling the output.

Lets slap Prisma into this puppy.

pnpm install typescript ts-node @types/node --save-dev

Prisma wants the ts-node package. We installed NextJS with Typscript so most of what we need is already there. Prisma asked for this so lets oblige.

Prisma is assuming that we are starting from scratch with Typescript. It asks that we create a new tsconfig.json file. But we already have one with quite a few settings thanks to NextJS.

Prisma wants the sourceMap and outDir added. The rest we already have. So add those to tsconfig.json in the compiler options and you’re good.

{
  "compilerOptions": {
    "sourceMap": true,
    "outDir": "dist"
  }
}

Looks like the app is still working which is good. Now we need to install Prisma.

pnpm install prisma --save-dev

I dont know how I feel about that but I’m going to let it live.

This next bit is interesting. It looks like we are going to build a local database in sqlite to test on. Sqlite is more like sqlVERYlite. But let’s see where this is headed.

pnpx prisma init --datasource-provider sqlite

It does its thing and spits out.


✔ Your Prisma schema was created at prisma/schema.prisma
  You can now open it in your favorite editor.

warn You already have a .gitignore file. Don't forget to add `.env` in it to not commit any private information.

Next steps:
1. Set the DATABASE_URL in the .env file to point to your existing database. If your database has no tables yet, read https://pris.ly/d/getting-started
2. Run prisma db pull to turn your database schema into a Prisma schema.
3. Run prisma generate to generate the Prisma Client. You can then start querying your database.

That sounds to me like… I can add an .env file, point it at some database that I have prebuilt and it will suck the schema into this prisma/schema.prisma file. Oh I hope so. For now, I’m going to drop in the SupaBase credentials and see if I can connect there.

Create a .env file in the root of your project. Add this to it.

DATABASE_URL="postgresql://johndoe:randompassword@dbhost:5432/mydb?schema=public"

Change the prisma/schema.prisma file to this.

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

I went and made a test table in supabase to see what happens. >.<

pnpm prisma db pull

Oh my stars! It worked! Prisma went and found my test table and added it to the schema.prisma file. It give a message to run prisma generate to generate Prisma Client.

pnpm install @prisma/client 

pnpm prisma generate

This adds a model of the tester table that I created in the Supabase ui. Let’s see if we can use it.

import { PrismaClient } from '@prisma/client'
const prisma = new PrismaClient()
